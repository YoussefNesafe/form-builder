# Socket supply-chain hardening — execution plan

_Date: 2026-07-20 · Status: DRAFT, awaiting owner approval · Executor: `orchestrate` agent_

Operationalizes the five long-term changes agreed from the Socket.dev reports
for the published npm package **`form-builder-nextjs`** (the CLI installer,
Unit B — `cli/`). Execution order is deliberate: **1 → 3 → 2 → 4 → 5**.

Read alongside `docs/plans/2026-07-18-packaging-publish-runbook.md` (§6 = the
CLI publish path) and `docs/adr/0003-packaging-split-distribution.md`.

## What Socket actually reported (baseline, v0.1.4)

- **Overall Socket score 79/100.** Sub-scores: Supply Chain 100, Vulnerability
  100, Quality 100, Maintenance 91, License 100 (MIT). **Zero CVEs.**
- **Package-level alerts** (on our own tarball): `URL strings` (doc/registry
  URLs in source), `AI-detected code anomaly` (heuristic flag on the CLI's
  dynamic fs copy/rewrite). Both benign, both expected for an installer.
- **Dependency-tab alerts** (transitive closure of the runtime libs the
  vendored code imports): `Unstable ownership` ×1 (**the one "high"** — an
  upstream dep changed publisher), `Network access`, `AI anomaly` ×2,
  `URL strings` ×13, `Environment variable access` ×6, `Minified code` ×6,
  `Unmaintained (>5yr)` ×4.

## Ground truth established before planning (verified 2026-07-20)

These change the scope of the items below — do not re-do work already shipped.

1. **Provenance is already live.** `npm view form-builder-nextjs dist` on
   `0.1.4` returns a SLSA-v1 `provenance` attestation **and** an npm registry
   signature. `.github/workflows/release-cli.yml` already declares
   `id-token: write` and auto-appends `--provenance` when the repo is public
   (keyed off `github.event.repository.private`). **Item 1 is therefore
   VERIFY + HARDEN, not build.**
2. **The vendored payload (`cli/vendor/`) is 100% first-party source** — a
   mirror of `form-builder/` + the 17 shadcn primitives, regenerated by
   `cli/scripts/vendor.mjs`. It is not third-party npm packages. Every
   dependency-tab alert therefore comes from the **runtime import closure**,
   not from our code. The top-level closure (`form-builder/package.json`):
   `@hookform/resolvers`, `class-variance-authority`, `clsx`, `cmdk`,
   `input-otp`, `libphonenumber-js`, `radix-ui`, `react-day-picker`,
   `react-phone-number-input`, `signature_pad`, `tailwind-merge`, `zustand`
   (+ peers: react, react-dom, react-hook-form, zod, date-fns, lucide-react).
   All top-level entries are modern/maintained — the 4 "unmaintained >5yr"
   flags are **deep transitive**, surfaced only by the Item 2 audit.
3. **Single maintainer** on npm: `youssef_nesafe <ynessafe@gmail.com>`.
4. **Publish auth is a long-lived `NPM_TOKEN` repo secret** (per the release
   workflow + runbook), not npm Trusted Publishing.

## Item 1 — Publish trust (provenance + token hardening)

**Goal:** lock in the trust signals; remove the long-lived-token attack surface.

**Current state:** provenance ✅ live on 0.1.4; npm signature ✅; publish still
uses a stored `NPM_TOKEN`.

**Steps**
1. **Verify + record** current state as the baseline (agent): confirm the
   attestation on the latest version, screenshot/paste `npm view … dist` into
   the audit trail.
2. **Migrate to npm Trusted Publishing (OIDC)** — owner links the GitHub repo +
   `release-cli.yml` workflow as a trusted publisher in the npm package
   settings; agent then removes the `NODE_AUTH_TOKEN`/`NPM_TOKEN` plumbing from
   the workflow and updates the runbook activation checklist. Net effect: no
   long-lived npm token exists to leak. **Owner gate.**
3. **Enable "require 2FA and disallow tokens"** on the npm package (owner), once
   Trusted Publishing covers CI so no automation token is needed.
4. Agent updates `docs/plans/2026-07-18-packaging-publish-runbook.md` §1/§6b to
   describe the token-free flow.

**Acceptance:** next publish is token-free (OIDC), carries provenance, and the
npm package shows 2FA-required. Runbook reflects it.

**Note:** provenance does **not** clear the "unstable ownership" dep alert —
that flag is on an upstream dependency, not our package. Item 2 owns that.

## Item 3 — Socket GitHub App (continuous monitoring + PR gate)

**Goal:** catch future dep drift / a malicious version swap before it ships,
automatically, on every PR.

**Steps**
1. **Owner installs the Socket for GitHub app** and grants it the
   `YoussefNesafe/form-builder` repo (free for OSS). **Owner gate.**
2. Agent adds a `socket.yml` at repo root tuning which alert categories block a
   PR vs. warn — start conservative: **block** on new `install scripts`,
   `shell access`, `network` on a *newly added* dep and malware signals; **warn**
   (don't block) on the existing heuristic noise (URL strings, env access,
   minified) so the gate isn't a false-positive wall on day one.
3. Agent documents the triage workflow (how to accept/ignore a known-benign
   alert) in a short `docs/` note so future PRs aren't blocked mysteriously.

**Acceptance:** a test PR that adds a throwaway dep triggers a Socket comment;
the existing baseline alerts do not block normal PRs.

## Item 2 — Dependency-closure audit (the transitive flags)

**Goal:** understand and shrink the dependency-tab alerts; decide accept vs.
replace per finding. **Deliverable is an audit doc, not code churn** — most
flags are benign traits of required libs.

**Steps**
1. Agent enumerates the full resolved runtime closure (top-level list above +
   transitive) and maps **each Socket alert instance to the offending
   package** — especially the **4 "unmaintained >5yr"** and the **1 "unstable
   ownership"**. Use `npm ls`, the lockfile, and the Socket dependency detail.
2. For each flagged package, classify:
   - **Accept + document** — required, maintained, trait is inherent
     (e.g. `libphonenumber-js` shipping URL strings, radix reading env in
     dev). Expected majority.
   - **Replace** — only if an unmaintained transitive dep has a drop-in
     maintained equivalent and the swap is low-risk. Do not force-replace a
     healthy top-level lib to chase a deep flag.
   - **Pin / constrain** — where a version bound would reduce drift risk.
3. Agent writes `docs/plans/2026-07-20-dependency-audit.md`: table of
   package → alert → verdict → action. Any actual dep change (replace/pin) is
   proposed there for a **separate follow-up PR**, not bundled blindly.

**Acceptance:** every dependency-tab alert has a named package and a written
verdict. The "unstable ownership" upstream package is identified and a
watch/accept decision is recorded.

## Item 4 — Ownership hygiene

**Goal:** reduce single-owner / account-compromise risk; make ownership legible.

**Steps**
1. **Owner:** confirm npm account 2FA is on (overlaps Item 1 step 3); decide
   whether to add a backup maintainer/owner on the npm package. **Owner gate.**
2. Agent adds a repo `CODEOWNERS` (`cli/**`, `form-builder/**`,
   `.github/workflows/release*.yml` → owner) so release-critical paths require
   owner review.
3. Agent writes a short **publish-policy** note (who may publish, that publishes
   go only through the tagged `release-cli.yml` workflow — never a local
   `npm publish` — so provenance is always attached).

**Acceptance:** `CODEOWNERS` guards release paths; publish policy documented;
2FA/backup-owner decision recorded.

## Non-goals / risks

- **Not** chasing Socket score to 100 — 79 with zero CVEs is healthy; the
  package/dep heuristic flags are inherent to an installer + a rich field lib.
- **Not** replacing healthy top-level deps to silence deep transitive flags.
- Trusted-Publishing migration (Item 1) must be verified with a `--dry-run`
  `workflow_dispatch` before the first token-free real publish, so a
  misconfigured OIDC link doesn't block a release.
- Socket PR gate (Item 3) starts in **warn-mostly** mode to avoid blocking
  legitimate PRs on day one; tighten later.
